# Session State And Storage Rediscovery

## Overview

Clauderfall is spending too much context on startup and continuity recovery relative to the value returned.

The immediate symptom is that a single startup-oriented request required reading multiple session handoffs, session lifecycle docs, persistence docs, runtime code, and stale filesystem artifacts just to determine what state was authoritative and what problem actually needed to be solved.

The problem is not just verbosity. The system currently appears to have conflicting or drifting authority boundaries between session handoff artifacts, session lifecycle docs, runtime storage, and stale persisted artifacts.

Part of the observed cost also came from skill behavior, not just persisted state shape.

The startup path appears willing to fan out into too much exploration too early instead of staying on the minimum authoritative read path needed to answer the operator's question.

## Problem Area: Startup Cost And Session Continuity

### Problem Statement

Recent-session continuity is not cheap or sharp enough.

Startup requires too much exploration to answer basic questions such as:
- what active work exists
- what persisted state is authoritative
- whether handoff data is still current
- whether the implementation still matches the documented model

This weakens the product goal of durable state with minimal token waste.

### Intended Outcomes

- Session startup should become materially cheaper in context cost.
- The authoritative session-state surface should be obvious.
- Continuation should not require reconciling multiple conflicting persistence stories.
- Handoff artifacts should stay bounded and continuation-focused rather than drifting into changelog prose.
- Startup skills should stay on the minimum read path needed for the question at hand.

### Constraints

- Preserve explicit operator-controlled persistence.
- Do not reintroduce implicit saves.
- Keep Discovery and Design workflow guarantees legible.
- Keep the product aligned with the single-senior-engineer usage model.

### Risks And Edge Cases

- Cleanup that only shrinks handoff prose may miss the deeper authority problem.
- Cleanup that only swaps storage technology may preserve an expensive startup model.
- The current repo may contain stale session artifacts that confuse startup and review.
- Session continuity and stage-artifact persistence may need different solutions rather than one shared substrate.
- Skill behavior may continue to waste context even if the persistence model improves.

## Problem Area: Storage Authority Drift

### Problem Statement

Clauderfall may no longer have one coherent persistence model.

The active docs describe layered session artifacts and document-first persistence, while the runtime uses SQLite-backed stores and the repo still contains older filesystem session artifacts.

This creates uncertainty about what should be read, what should be trusted, and what needs to be maintained.

### Intended Outcomes

- One persistence story should be authoritative for session continuity.
- Docs and runtime should describe the same model.
- Obsolete persisted state should stop consuming attention during startup and diagnosis.

### Open Questions

- Is this rediscovery scoped only to recent-session continuity, or does it include the broader persistence substrate for Discovery and Design artifacts too?
- Should session continuity and stage artifact persistence remain coupled at all?

## Current Readiness

Not ready for Design.

The main blocker is scope ambiguity: it is still unclear whether the problem to frame is:
- session continuity cost and handoff shape only, or
- the broader storage substrate and authority model across Clauderfall runtime state.
