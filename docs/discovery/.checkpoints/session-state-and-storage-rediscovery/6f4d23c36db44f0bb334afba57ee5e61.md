# Persistence Substrate Rediscovery

## Overview

Clauderfall may no longer have one coherent persistence model for its authoritative runtime state.

The active docs describe a document-first and layered artifact model, while the runtime uses SQLite-backed stores, and the repo still contains older filesystem session artifacts that no longer clearly match the implementation.

This creates uncertainty about what should be read, what should be trusted, and what needs to be maintained.

This Discovery is scoped to the persistence substrate first.

It does not yet try to solve session continuity behavior, startup skill behavior, or handoff content quality except where those expose substrate-level authority problems.

The main product pressure is authority, not just implementation taste.

The operator wants persisted authoritative state to live in the repo, with metadata living alongside the corresponding readable artifact data rather than hidden inside a separate database.

Possible team use now matters as well.

If Clauderfall is used by a small team, one operator's Claude may need to inspect another operator's structured metadata directly from the repo.

That raises the bar for inspectability, co-location, and shared authority.

## Problem Area: Persistence Authority Drift

### Problem Statement

Clauderfall appears to have conflicting persistence stories across docs, runtime implementation, and leftover on-disk artifacts.

The current state makes it harder to answer basic questions such as:
- what persisted state is authoritative
- which substrate owns current runtime truth
- whether filesystem artifacts, SQLite state, or both must be maintained
- whether session state and Discovery/Design artifact state should share one persistence model at all

This creates design drift, review friction, and extra exploration cost.

### Intended Outcomes

- One persistence story should be authoritative for Clauderfall runtime state.
- Authoritative state should live in the repo rather than in a separate hidden database.
- Structured metadata should live with the corresponding artifact data, even if stored in an adjacent or hidden repo path.
- Docs and runtime should describe the same substrate.
- Obsolete persisted state should stop consuming attention during diagnosis and startup.
- It should be explicit whether session continuity and stage artifacts share one persistence approach or intentionally diverge.

### Constraints

- Preserve explicit operator-controlled persistence.
- Keep Discovery and Design workflow guarantees legible.
- Preserve deterministic backend validation and state transitions where they exist.
- Do not let cleanup silently weaken artifact readability or structured authority.
- Keep persisted state inspectable enough for small-team collaboration, not just single-user local recovery.

### Risks And Edge Cases

- Forcing one shared substrate across all artifact classes may preserve unnecessary coupling.
- Splitting substrates carelessly may fragment the product and make state harder to reason about.
- Migrating away from the current implementation may leave stale state or dual-write confusion behind.
- Updating docs without clarifying authority boundaries would only rename the inconsistency.
- Repo-visible metadata may increase churn or review noise if the artifact shape is not kept bounded.

## Problem Area: Session And Stage Coupling

### Problem Statement

Session continuity state and stage artifact state may be coupled more tightly than the product actually needs.

Discovery and Design artifacts need readable documents plus structured state with checkpoint semantics.

Session continuity may have different durability, visibility, and history needs.

If these are forced into one substrate for conceptual neatness, the persistence model may become harder to reason about and more expensive to maintain.

At the same time, if the product may be used by a small team, the argument for repo-visible, inspectable state may extend beyond just design documents.

### Intended Outcomes

- The product should be clear about whether session continuity and stage artifacts are one storage problem or two related ones.
- Any shared substrate should be justified by concrete product needs rather than architectural preference.
- Any split substrate should still leave authority and inspection boundaries obvious.
- The product should be explicit about which artifact classes must be inspectable and shareable through the repo for team use.

### Open Questions

- Should Discovery and Design artifact persistence stay coupled to session-state persistence at all?
- Is SQLite still justified anywhere in the product boundary?
- Are leftover filesystem session artifacts historical debris, migration residue, or an intended future direction?
- If the product expands from one engineer to a small team, which artifact classes actually need repo-visible metadata sharing?

## Current Readiness

Not ready for Design.

The persistence problem is now clearer, but one boundary still blocks readiness: whether repo-visible authority is required for all persisted artifact classes, or only for Discovery and Design artifacts.
