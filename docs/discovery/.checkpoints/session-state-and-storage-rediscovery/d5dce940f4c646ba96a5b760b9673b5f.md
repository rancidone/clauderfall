# Persistence Substrate Rediscovery

## Overview

Clauderfall may no longer have one coherent persistence model for its authoritative runtime state.

The active docs describe a document-first and layered artifact model, while the runtime uses SQLite-backed stores, and the repo still contains older filesystem session artifacts that no longer clearly match the implementation.

This creates uncertainty about what should be read, what should be trusted, and what needs to be maintained.

This Discovery is scoped to the persistence substrate first.

It does not try to solve session continuity behavior in full.

The main product pressure is authority, not just implementation taste.

The operator wants authoritative persisted state to live in the repo, with metadata living alongside the corresponding artifact data rather than hidden inside a separate database.

Possible team use now matters as well.

If Clauderfall is used by a small team, one operator's Claude may need to inspect another operator's structured metadata directly from the repo.

That raises the bar for inspectability, co-location, and shared authority.

At the same time, session continuity is better framed as a personal developer concern rather than a shared team artifact class.

Longer-lived shared history can often be carried by ordinary git history rather than by a product-managed session archive.

What remains for session continuity is a bounded handoff artifact for work that exceeds one context window.

That handoff should stay small, be trimmed over time rather than accumulate, and may disappear entirely when there is nothing meaningful to carry forward.

For now, no additional first-class persisted repo artifact class is required beyond Discovery and Design.

Personal TODOs are better treated as content inside Discovery, Design, or a bounded handoff rather than as a separate persistence substrate.

The product position is now explicit: SQLite should be removed from the authoritative persistence substrate.

Clauderfall is intended to help an LLM work with inspectable project artifacts, and the LLM is already naturally good at reading and writing files.

A separate SQLite authority layer works against that purpose by hiding state outside the repo and creating a second persistence model that the LLM does not naturally inspect.

## Problem Area: Persistence Authority Drift

### Problem Statement

Clauderfall appears to have conflicting persistence stories across docs, runtime implementation, and leftover on-disk artifacts.

The current state makes it harder to answer basic questions such as:
- what persisted state is authoritative
- which substrate owns current runtime truth
- whether filesystem artifacts, SQLite state, or both must be maintained
- which artifact classes actually need first-class persisted repo state

This creates design drift, review friction, and extra exploration cost.

### Intended Outcomes

- One persistence story should be authoritative for Clauderfall runtime state.
- Authoritative shared state should live in the repo rather than in a separate hidden database.
- Structured metadata should live with the corresponding artifact data, even if stored in an adjacent or hidden repo path.
- SQLite should no longer be part of the authoritative product persistence model.
- Docs and runtime should describe the same substrate.
- Obsolete persisted state should stop consuming attention during diagnosis and startup.
- The product should avoid inventing extra first-class artifact classes without a clear need.

### Constraints

- Preserve explicit operator-controlled persistence.
- Keep Discovery and Design workflow guarantees legible.
- Preserve deterministic backend validation and state transitions where they exist, even if the storage substrate changes.
- Do not let cleanup silently weaken artifact readability or structured authority.
- Keep shared persisted state inspectable enough for small-team collaboration, not just single-user local recovery.

### Risks And Edge Cases

- Splitting substrates carelessly may fragment the product and make state harder to reason about.
- Migrating away from the current implementation may leave stale state or dual-write confusion behind.
- Updating docs without clarifying authority boundaries would only rename the inconsistency.
- Repo-visible metadata may increase churn or review noise if the artifact shape is not kept bounded.
- Replacing SQLite badly could regress validation or transition correctness if deterministic runtime behavior is tied too tightly to the current store implementation.

## Problem Area: Shared Artifacts Versus Personal Continuity

### Problem Statement

Clauderfall may be mixing two different kinds of persisted state.

Discovery and Design artifacts look like shared, reviewable, repo-visible project artifacts.

Session continuity looks more like a personal, bounded handoff mechanism for long-form work that exceeds one context window.

If these are treated as the same persistence problem, the product may overbuild continuity history or force personal carry-forward state into shared project storage rules that it does not need.

### Intended Outcomes

- The product should be explicit that Discovery and Design artifacts are shared repo-visible artifacts.
- The product should be explicit that session continuity is a separate personal handoff problem unless proven otherwise.
- Git history should remain a valid source of longer-lived project history instead of duplicating that role inside session-state persistence.
- Any persisted handoff artifact should stay bounded and should not accumulate into a second historical archive by default.
- Personal TODO-like planning should remain content within existing artifact types rather than becoming a new persisted substrate.

### Open Questions

- Are leftover filesystem session artifacts historical debris, migration residue, or an intended future direction?
- Is a bounded personal handoff artifact enough for session continuity, with no product-managed shared continuity history beyond git?

## Current Readiness

Close to ready for Design.

The main remaining gap is cleanup and interpretation of leftover session artifacts from the hybrid period, plus whether the bounded-handoff model for session continuity is settled enough to carry forward into its separate Discovery.
